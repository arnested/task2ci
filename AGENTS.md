# AGENTS.md

Guidelines, guards, and conventions for agents working in this repository.
This file captures decisions made over the lifetime of the project; treat
them as established unless the user redirects.

## What this project is

`task2ci` is a single-binary Go tool that generates GitHub Actions workflows
from a project's `Taskfile.yaml`, driven by `@ci:` annotations on tasks and a
`.task2ci.yaml` config. The README is the user-facing intro; this file is
about how to work _on_ the codebase.

## Preflight (run this before saying a change is done)

The default Taskfile task runs the full preflight in parallel:

```sh
go tool task
```

That covers: `build`, `test`, `vet`, `lint` (golangci-lint), `yamllint`,
`markdownlint`, `check` (drift detection), `schema-check`. All must be green.
The same task runs via the tracked pre-commit hook at `.githooks/pre-commit`.

Run individual steps with `go tool task <name>` when iterating.

## Generated files — do not hand-edit

- `.github/workflows/*.yaml` — generated by `task2ci` from `Taskfile.yaml` and
  `.task2ci.yaml`. The header says so. Edits will be reverted on the next
  generation and `-check` will fail in CI.
  - `.gitattributes` marks them `linguist-generated=true`.
  - To change them: edit `Taskfile.yaml` and/or `.task2ci.yaml`, run
    `go tool task generate`, commit the resulting diff.

## Tool dependencies

External tooling is intentionally pinned via Go's `tool` directive in
`go.mod`. Invoke via `go tool <name>` rather than assuming a system binary:

| Tool | How |
|------|-----|
| `task` (go-task) | `go tool task` |
| `golangci-lint` | `go tool golangci-lint` |
| `yajsv` (JSON Schema validator) | `go tool yajsv` |

Don't `go install ...` system-wide. If a new linter/checker is needed,
add it with `go get -tool <module>` and use `go tool` to invoke it.

`yamllint` and `markdownlint-cli` are exceptions — they're not Go and live
outside go.mod. `yamllint` is assumed installed (system package).
`markdownlint-cli` is invoked via `npx --yes markdownlint-cli@0.41.0` because
versions ≥ 0.42 require Node 20+ and the current dev environment is Node 18.
If the runtime Node version is upgraded, drop the pin.

## Code conventions

### Single file (`main.go`)

The whole tool lives in `main.go`. Don't split into packages unless the
file grows beyond what's reasonable to navigate. Tests are in `main_test.go`.

### YAML output (yaml.v3 quirks)

The generator uses `gopkg.in/yaml.v3`. Several things matter:

1. **`yaml.Node.Encode(v)` strips HeadComment from `MarshalYAML` results.**
   If a nested type's `MarshalYAML` returns a `*yaml.Node` with a
   `HeadComment`, that comment is lost when the parent is encoded via
   `.Encode`. Workaround: build the parent's node tree manually instead of
   `.Encode`-ing the value. See `Job.MarshalYAML` and `Jobs.MarshalYAML`.
2. **Blank lines come from `HeadComment` starting with `"\n"`.** yaml.v3 has
   no native blank-line emit. Setting `HeadComment = "\n"` produces a blank
   line above the node; `"\n<text>"` adds the blank line then a `# <text>`
   comment.
3. **yaml.v3 emits indent on blank lines**, which `yamllint` flags as
   trailing whitespace. We post-process generated output via
   `stripTrailingWhitespace` before writing.
4. **`on:` is quoted by default struct encoding** because YAML 1.1 treats it
   as a boolean. We build the `Workflow` node manually via `MarshalYAML` so
   `on:` stays unquoted (GitHub Actions convention).

### Annotation syntax

Tasks opt into CI with a comment on the task key or any inner field:

```yaml
# @ci: <tag>              # tag only
# @ci: <tag> | <step name>  # tag plus display-name override
```

The `|` was chosen over `:`, `->`, and quoted-string because it has the
simplest parser (single `strings.Cut`) and no conflicts with names that
contain colons, commas, quotes. Don't change the separator without
discussing.

Step name resolves in this order: annotation override → task `desc:` → task
name. Don't add comment-from-desc behavior back (it was removed because the
desc is more useful as the visible step name).

### Schema (`task2ci.schema.json`)

The schema is the source of truth for `.task2ci.yaml` structure. It is:

- Embedded into the binary via `//go:embed` and used by `task2ci -check` to
  validate the live config (`validateConfigAgainstSchema`).
- Referenced by the project's own `.task2ci.yaml` for editor validation via
  yaml-language-server.
- Validated end-to-end in CI by the `schema-check` task (yajsv).

When changing config structure: update `Config`/`TagSpec` in `main.go`,
update `task2ci.schema.json`, update the schema's pattern/required rules,
update the `initTemplate`, run `go tool task` to confirm everything stays
in sync.

## Tests

- Pure-helper tests (`hasToolDirective`, `parseAnnotation`, `taskDesc`,
  YAML marshaling) call functions directly.
- Integration tests run the built binary against fixture directories under
  `t.TempDir()`. The binary is built once per `go test` invocation via a
  `sync.Once`-guarded helper (`buildBinary`).
- New behavior should get both a unit test (if the helper is pure) and an
  integration test covering the user-facing CLI surface.
- Fixtures embed the yaml-language-server schema-ref comment via the shared
  `schemaRef` const — keep new fixtures consistent.

## CLI surface

```text
task2ci             # generate
task2ci -check      # validate config against schema, then check for drift
task2ci -init       # write starter .task2ci.yaml (refuses to overwrite)
```

Keep the flag set minimal. Adding a flag deserves a discussion about whether
it belongs as a subcommand or a config field instead.

## Style preferences observed in this project

- **Generated workflows are formatted for human readability:** blank lines
  between top-level keys, between jobs, and between steps; descriptive
  `name:` per step. Don't optimize for terseness; the diff matters less than
  the CI-log readability.
- **Tag defaults are friendly:** missing `workflow:` defaults to
  `"taskfile"`, missing `job:` defaults to the tag name. Only `runs-on` is
  required. Preserve this — adding required fields breaks backwards-compat.
- **Fatal errors over warnings for ambiguous config:** conflicting
  `runs-on` for the same `(workflow, job)` is fatal, not warned-with-pick.
  Unknown tag references are warnings (task is skipped, generation
  continues).
- **One-off task in main, not a refactor:** the user has pushed back on
  premature splitting. Keep changes local unless explicitly invited to
  restructure.

## Gotchas

- **Emacs lock files** (`.#filename`) appear and disappear as files are
  opened in the IDE. They're symlinks to nonexistent targets and break
  tools that try to follow them. `.yamllint` ignores them; `.gitignore`
  excludes them. Don't be alarmed if `git status` shows them temporarily.
- **`runs-on` quote sensitivity:** the `Workflow.On` field becomes the
  `on:` key (sequence) in workflow YAML; that's the only place where YAML
  1.1 booleans bite us. Tested in `TestWorkflowEncodeStructure`.
- **Tool-dependency detection** is positional in `go.mod` — both the
  `tool github.com/...` single-line form and the `tool ( ... )` block form
  are supported. See `hasToolDirective`. Don't change to a regex; the
  current line walker handles trailing `//` comments correctly.

## When in doubt

- Run `go tool task`. If it passes, the change is likely fine.
- Run `./task2ci` then `./task2ci -check` to confirm self-CI is consistent.
- The user has been clear about wanting concise, focused changes — don't
  bundle drive-by refactors with feature work.
